Foundation Infrastructure · Microsoft 365

Microsoft 365

Tenant configuration, licensing, and core app deployment. M365 is the workspace layer for every RTBoss deployment — Teams is the human interface, SharePoint is the document layer, Loop is the living knowledge base, and the admin centre governs identity and access across everything.

Role in RTBoss
Every human worker in every Mini-OS operates through M365. Approval cards surface in Teams. Documents live in SharePoint. Tasks track in Planner and To Do. Loop holds the knowledge base. M365 is not optional — it is the human layer of RTBoss.
Prerequisites before starting this section
· Domain ownership verified
· Billing account established
· Admin credentials secured in password manager
Feeds into
All Mini-OS (human interface layer) Productivity OS (Tasks, Focus Blocks, Loop KB) Strategy OS (Viva Goals, OKR tracking) HR OS (Viva, onboarding journeys)
Tenant Setup & Licensing
Key Decisions
Tenant domain
Primary domain registered and verified. Custom domain preferred over .onmicrosoft.com for all client-facing communication.
Licensing tier
Microsoft 365 Business Premium recommended as baseline — includes Intune, Defender, Entra ID P1. Upgrade to E3/E5 for enterprise clients requiring Purview or advanced compliance.
Data residency
Australia region selected at tenant creation. Cannot be changed post-creation. Critical for AU data sovereignty requirements.
Configuration Areas
Tenant domain verification
License assignment model (per-user vs shared)
Admin roles defined (Global, Security, Compliance, Helpdesk)
MFA enforced for all users from day one

Business Application Layer · Dynamics 365

Dynamics 365

Business Central, Contact Center, Customer Service, Field Service, and Sales. Dynamics 365 is the system of record for business transactions — every invoice, ticket, deal, work order, and contact center interaction lives here. Mini-OS Digital Assets are built on the Dynamics data model, extended via Dataverse.

Role in RTBoss
Dynamics 365 is where business operations are executed and recorded. Finance OS runs on Business Central. Sales OS runs on D365 Sales. Services OS and MIP run on Customer Service. LiveOps runs on D365 Contact Center. Intelligent Buildings field operations run on Field Service. All transaction data flows to the Digital Twin.

Licensing

Digital Asset Layer · Dataverse

Dataverse

The unified data layer for all RTBoss Digital Assets. Every CLIENT, INVOICE, EMPLOYEE, CONTRACT, CAMPAIGN, and OPPORTUNITY record lives in Dataverse — structured, governed, and accessible to every Genie and Stream in the Agentic Factory. Dataverse is what makes Digital Assets persistent across the entire RTBoss loop.

Role in RTBoss
Dataverse is the single source of truth for all Digital Assets. When a Genie reads client history, when a Stream updates an asset state, when RTBI queries current asset status — all of it reads from and writes to Dataverse. The schema design here determines what every downstream component can see.
Prerequisites before starting this section
· Power Platform environments created
· Security model designed
· Naming convention agreed before any tables created
Feeds into
All Mini-OS (Digital Asset storage) All Value Engines (cross-OS asset reads) RTBI (Analytics queries Dataverse for current asset state)
Environment & Governance
Key Decisions
Environment strategy
Dataverse environments map to Power Platform environments. Production Dataverse is never used for development. Each environment has its own Dataverse instance.
Security model
Column-level security for sensitive fields (salary, health data, financial data). Business unit hierarchy matches organisational structure. Table-level permissions enforced via security roles.
Configuration Areas
Dataverse environment creation
Security roles per Mini-OS
Business unit structure
Audit logging enabled from day one
Data loss prevention policies
⧖ STEP-BY-STEP PENDING
Step-by-step: Power Platform admin centre, Dataverse provisioning, security role creation, DLP policy configuration
Digital Asset Schema Design
Key Decisions
Table naming convention
Prefix all custom tables with 'neit_' (New Era IT) to distinguish from standard Dataverse tables. Table names match Digital Asset names: neit_invoice, neit_employee, neit_contract etc.
Core vs domain tables
Core tables (shared across Mini-OS): neit_fact, neit_dimension, neit_persona. Domain tables: one set per Mini-OS containing that domain's Digital Assets.
Relationship design
Facts link to Digital Assets via lookup columns. Digital Assets link to each other where business relationships exist (e.g. INVOICE → VENDOR → CONTRACT).
Configuration Areas

Sales OS tables: neit_lead, neit_opportunity, neit_account, neit_deal
Services OS tables: neit_engagement, neit_sla, neit_milestone, neit_client
Remaining Mini-OS tables: follow same pattern

Agentic Factory Orchestration · Power Platform

Power Platform

Power Automate, Power Apps, and Copilot Studio. Power Platform is the orchestration layer of the Agentic Factory — Streams are built in Power Automate, approval interfaces are built in Power Apps, and Hybrid Worker Genies are configured in Copilot Studio. This is where the RTBoss loop is wired together.

Role in RTBoss
Power Platform connects everything: Dataverse Digital Assets, Dynamics 365 business records, Teams human interfaces, Azure AI Foundry Genies, and external APIs. Every Stream is a Power Automate flow. Every Hybrid Worker approval card is a Power Apps or Teams Adaptive Card. Every Genie conversation surface is a Copilot Studio agent.
Prerequisites before starting this section
· Dataverse environments live
· DLP policies defined before first flow is built
· Azure AI Foundry provisioned (for Copilot Studio → Foundry connection)
Feeds into
All Mini-OS (Streams built in Power Automate) All Hybrid Worker Genies (Copilot Studio) Finance OS approval workflows HR OS onboarding flows Legal OS deadline monitoring
Environment & Governance
Key Decisions
DLP policies
Data Loss Prevention policies defined before any flows are built. Classify connectors: Business (approved), Non-Business (restricted), Blocked. Prevents flows from sending data to unapproved external services.
Environment strategy
Default environment never used for production. Dedicated environments: Production, UAT, Dev. Solution-based ALM (Application Lifecycle Management) for all flows — no ad-hoc flows in production.
Managed environments
Managed Environments enabled in production — adds governance controls, usage analytics, and maker onboarding. Required for ISO 27001 compliance posture.
Configuration Areas
DLP policy creation (Business/Non-Business/Blocked connectors)
Managed Environments activation
Environment-level security roles
Power Platform CoE Starter Kit deployment (optional but recommended)
Maker access policies

Power Automate — Stream Deployment
Key Decisions
Solution-based flows
All production flows built inside solutions. Solutions are exported from Dev, imported to UAT for testing, promoted to Production. Never build flows directly in production.
Connection references
Connection references used instead of hardcoded connections — allows solution transport between environments without breaking connections.
Error handling
Every flow includes error handling scope. Failed runs write a Fact to Dataverse: flow_failed · error_message · affected_stream. Digital Twin captures all failures.

Intelligence & Physical Layer · Azure

Azure

AI Foundry, Microsoft Fabric, Cosmos DB, IoT Hub, and supporting services. Azure is where the intelligence happens — Genies run in AI Foundry, the Digital Twin lives in Cosmos DB, Unified Data flows through Fabric, and physical world Facts enter via IoT Hub.

Role in RTBoss
Azure is the backend intelligence layer that Power Platform and Dynamics 365 both depend on. Without Azure: no Digital Twin, no real-time Unified Data, no LLM-powered Genies, no IoT-driven physical Facts. Every RTBoss deployment of any scale requires Azure services.
Prerequisites before starting this section

· Entra ID tenant connected
· Power Platform environments live (for AI Foundry → Copilot Studio connection)

Feeds into

Key Decisions
Subscription model
Separate Azure subscriptions for Production and Non-Production recommended for larger deployments. Single subscription with resource group separation acceptable for smaller deployments.

Australia East primary region. Australia Southeast for geo-redundant backups. No data leaves Australia for AU data sovereignty.

Resource naming: rtbos-{env}-{component}-{region}. e.g. rtbos-prod-foundry-aue. Enforced via Azure Policy.
Cost management
Budget alerts configured from day one. Tags applied to all resources: environment, mini-os, pillar, cost-centre. Cost tracking feeds Finance OS RTBI Intelligence Genie.

Azure Cosmos DB — Digital Twin
Key Decisions
API choice
NoSQL API for the Digital Twin. Document model maps naturally to Fact records — each document is one Fact with all enrichment data.
Append-only design
Digital Twin is append-only. No document is ever updated or deleted — only new documents added. Immutability enforced via Cosmos DB stored procedures and role permissions. No write role includes delete.
Partition strategy
Partition key: Mini-OS domain + month (e.g. 'finance-2026-02'). Enables efficient time-range queries for RTBI and Digital Twin replay.
Backup
Continuous backup enabled. Point-in-time restore to any second in the last 30 days. Cross-region replication to Australia Southeast.
Configuration Areas
Cosmos DB account creation (AU East)
Database and container creation per Mini-OS
Partition key configuration
Indexing policy (time-series optimised)
Append-only role assignment
Backup and replication configuration
Diagnostic logs to Log Analytics

RMM & Auto-Remediation · Zero Touch AI

Zero Touch AI

The RMM platform that turns every device, server, and endpoint into a source of Facts for the RTBoss loop. Zero Touch AI agents capture telemetry, detect anomalies, and execute auto-remediation — feeding the MIP Mini-OS with a continuous, structured stream of IT infrastructure events.

Role in RTBoss
Zero Touch AI is the fact-capture layer for MIP. Without it, MIP has no visibility into client infrastructure. Every monitoring alert, patch event, performance threshold, and auto-remediation action it generates becomes a Fact that enters the RTBoss loop — enriched, routed, actioned, and logged.

Monitoring Policies
Key Decisions
Policy inheritance
Global policies apply to all devices. Site-level overrides for client-specific thresholds. Device-level overrides for exceptions. Hierarchy: device > site > global.
Alert thresholds
CPU (>85% sustained 5min), Memory (>90%), Disk (>85% used), Disk health (SMART failure), Service failure, Patch compliance (<95%). These thresholds generate monitoring_alert Facts in RTBoss.
Alert suppression
Maintenance windows configured to suppress non-critical alerts during scheduled maintenance. Prevents alert storms from generating false Facts.

Security Foundation · Zero-Trust & Compliance

Security & Compliance

Entra ID, Intune, Defender, and Purview. The security layer that underpins every RTBoss deployment. Zero-trust architecture, Australia data residency, ISO 27001 alignment, and complete audit trail from every Fact to every Digital Asset. Security is not a layer added later — it is configured before the first user is provisioned.

Role in RTBoss
Security is the foundation everything else sits on. Identity (Entra ID), device compliance (Intune), threat protection (Defender), and data governance (Purview) must all be configured before any Mini-OS, Genie, or Stream goes live. Every security event is itself a Fact — breach_detected, policy_violation, access_anomaly — that enters the RTBoss loop.
Prerequisites before starting this section
· This section has no prerequisites — it IS the prerequisite for everything else. Configure security before provisioning users, before deploying apps, before building any Stream.
Feeds into

Entra ID — Identity Foundation
Key Decisions
Conditional Access
Baseline Conditional Access policies from day one: require MFA for all users, block legacy auth, require compliant device for sensitive data access, restrict access to AU locations for admin accounts.
Privileged Identity Management
PIM enabled for all admin roles. No standing admin access — just-in-time elevation with approval and time limit. Every elevation is a Fact in the Digital Twin.
External identities
Guest users for client collaboration via B2B. Guest access scoped to specific Teams and SharePoint sites only — never to Dataverse or Azure resources.
Configuration Areas
MFA enforcement (Authenticator app, not SMS)
Conditional Access baseline policies
Named locations (AU offices, trusted IPs)
PIM role activation policies
Group structure (security groups for Mini-OS access)
Emergency access accounts (break-glass)

Intune — Device Compliance
Key Decisions
Compliance policies
Minimum compliance: OS version current, BitLocker/FileVault enabled, antivirus active, screen lock configured. Non-compliant devices blocked from corporate resources via Conditional Access.
Configuration profiles
Standard configuration profiles pushed to all managed devices: security baseline, Edge configuration, OneDrive known folder move, Windows Update rings.
BYOD policy
BYOD enrolled in Intune MAM (Mobile App Management) without MDM — corporate data in M365 apps is protected without full device management. Personal devices never get full MDM unless contractually required.
Configuration Areas
Compliance policy (Windows, macOS, iOS, Android)
Configuration profiles (security baseline)
Windows Update rings (pilot, broad, deferred)
BitLocker encryption policy
App protection policies (MAM for BYOD)
Autopilot deployment profile for new devices

Microsoft Defender — Threat Protection
Key Decisions
Defender for Endpoint
P2 licence for all managed devices. Attack surface reduction rules enabled. Tamper protection on. Defender alerts become security_alert Facts in RTBoss — routed to MIP Triage Genie.
Defender for Identity
Monitors Entra ID for identity-based attacks — lateral movement, pass-the-hash, DCSync. Alerts surface to LiveOps command center as security_incident Facts.
Defender for Cloud Apps
CASB layer. Monitors cloud app usage, flags shadow IT, enforces session controls for sensitive apps. Unusual access patterns generate access_anomaly Facts.
Configuration Areas
Defender for Endpoint onboarding (via Intune)
Attack surface reduction rules (audit first, enforce after baseline)
Defender for Identity sensor deployment
Defender XDR integration (unified SOC view)
Alert suppression rules (reduce noise)
RTBoss webhook: Defender alert → Power Automate → security_alert Fact

Microsoft Purview — Data Governance
Key Decisions
Sensitivity labels
Label taxonomy: Public, Internal, Confidential, Highly Confidential. Labels applied to M365 content, emails, and meetings. Confidential and above requires encryption.
Retention policies
Retention per content type: financial records (7 years AU), HR records (7 years), legal documents (contract term + 7 years), general business (3 years). Enforced via Purview retention labels.
Audit log
Unified audit log enabled and retained for minimum 12 months (90 days default — extend via Purview Audit Premium). Every admin action, every file access, every permission change is logged. This is the compliance Digital Twin.
Configuration Areas
Sensitivity label taxonomy and policies
Auto-labelling policies (sensitive data types)
Retention label and policy creation
Audit log retention extension
Data loss prevention policies (credit card, TFN, health data)
Communication compliance policies (if required)
eDiscovery configuration
