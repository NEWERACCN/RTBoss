Canonical Deployment Registry

Agentic Factory Library

The single source of truth for every Digital Worker across the RTBoss platform. Genies, Streams, Skills, Advisors, Flows, Code modules, and composite Digital Workers are all registered here. Agents post to this registry automatically on first successful deployment — a human confirms to promote status to deployed.

??? Dataverse · ENT-004 Digital Worker Registry — master record store ?? GitHub · neit-rtboss — system prompts, configs, skill definitions ?? Copilot Studio — Genie + Advisor runtime ? Power Automate — Stream + Flow runtime
???
One master database. Every Digital Worker type — regardless of what it is or where it runs — has a single record in Dataverse ENT-004. The dw_type field distinguishes what it is. The runtime field says where it runs. GitHub stores the source. Copilot Studio or Power Automate is the execution environment. Dataverse is the registry. Nothing is deployed without a registry record.
?
Auto-post on publish. When a Digital Worker completes its first successful run, it writes a draft record to ENT-004 with status building. A Tech Lead or CSM reviews and promotes to deployed. Status advances to proven after 30 days of successful operation across at least one client. No manual data entry required — the agent registers itself.
ENT-004 · Digital Worker Registry — Dataverse Schema
ENT-004
Digital Worker Registry
Master record for every deployed or planned Digital Worker. Written by agents on first run, confirmed by humans. The registry this page reads from.
dw_id
Text · Primary Key · Auto-assigned
Unique identifier. Prefix determines type: GNI-, STR-, SKL-, ADV-, FLW-, COD-, DWK-. Sequential within type e.g. GNI-001, GNI-002.
dw_name
Text · Required
Canonical name matching the definition in Section 09 · Used as display label across all registry views.
dw_type
Choice · Required
Genie / Stream / Skill / Advisor / Flow / Code / Worker. Determines which group the record appears in on the Registry tab.
dw_status
Choice · Required · Default: planned
planned ? building ? deployed ? proven. Agent auto-sets to building on first run. Human promotes to deployed. System promotes to proven after 30 days.
dw_version
Text · Required · Default: 1.0
Semantic version of the current deployment. Bumped on any system prompt, config, or flow change. Matches GitHub tag.
mini_os
Text · Required
Owning Mini-OS e.g. Finance OS, HR OS, Strategy OS. Use "All" for cross-cutting workers. Matches Mini-OS definitions in Section 03 ·
owning_system
Text · Optional
The System within the Mini-OS that governs this worker e.g. AP System, Client Context System. Blank if cross-system.
runtime
Choice · Required
Where the worker executes: Copilot Studio / Power Automate / Azure Function / GitHub Actions / Manual. Determines which runtime reference fields are populated.
github_repo
Text · Optional
GitHub repository name e.g. neit-rtboss or client-specific repo. Source of truth for all configs and system prompts.
github_path
Text · Optional
Path within the repo to the worker's definition folder e.g. /genies/GNI-001-invoice-approval/ or /skills/SKL-001-pdf-extract/.
cs_agent_name
Text · Optional
Copilot Studio agent name if runtime = Copilot Studio. Used by other agents to invoke this worker via the Copilot Studio API.
pa_flow_name
Text · Optional
Power Automate flow display name if runtime = Power Automate. Streams and Flows populate this field.
clients
Text · Multi-value · Optional
Comma-separated client codes with deployment status e.g. EMS:deployed, RDM:building, RLAW:planned. Updated by the worker on each new client deployment.
last_published
DateTime · Auto
Timestamp of the last deployment event — version bump, new client deployment, or status change. Set by the agent, not by humans.
published_by
Text · Auto
ID of the agent or human who triggered the last status change. Format: agent ID (e.g. GNI-003) or Entra UPN for human confirmations.
notes
Text (Long) · Optional
Free-text notes — known limitations, client-specific config overrides, deprecation notices, or links to related NCRs.
Auto-Post Payload — What an Agent Writes on First Successful Run
??
Registration flow: On first successful completion, the agent invokes SKL-015 d365_write with the payload below. Status is set to building. A notification fires to PRS-006 (Tech Lead) via SKL-011. Tech Lead reviews and updates status to deployed via the confirmation UI. After 30 days of clean operation, the system auto-promotes to proven.
// Auto-post payload written by agent on first successful run { "entity": "ENT-004_DigitalWorkerRegistry", "action": "create_draft", "payload": { "dw_id": "{{auto_assigned}}", // assigned by Dataverse on create "dw_name": "{{canonical_name}}", // from system prompt header "dw_type": "{{Genie|Stream|Skill|...}}", // declared in config "dw_status": "building", // always building on auto-post "dw_version": "{{version_from_github_tag}}", // e.g. "1.0" "mini_os": "{{owning_mini_os}}", // declared in config "owning_system": "{{owning_system}}", // declared in config "runtime": "{{runtime_environment}}", // Copilot Studio | Power Automate | etc "github_repo": "{{repo_name}}", // from deployment config "github_path": "{{repo_path}}", // from deployment config "cs_agent_name": "{{cs_agent_name|null}}", // null if not Copilot Studio "pa_flow_name": "{{pa_flow_name|null}}", // null if not Power Automate "clients": "{{client_id}}:building", // first client, status building "last_published": "{{utc_timestamp}}", // auto "published_by": "{{self_dw_id}}", // the agent registering itself "notes": "Auto-registered on first run. Awaiting Tech Lead confirmation." }, "notify": { "persona": "PRS-006", "channel": "teams", "message": "New Digital Worker registered — {{dw_name}}. Review and confirm deployment status." } }
