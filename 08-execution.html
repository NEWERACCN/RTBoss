Execution & Tracking · Architecture

Tracking System Overview

This section defines the architecture and design of the RTBoss tracking system — what it is, where it lives, how it works, and how each client gets their own instance. Not the live tracker itself, but the blueprint that governs every tracker built.

?
RTBoss eats its own cooking. The tracking system is built on RTBoss principles — every implementation step is a Fact, every agent deployment is a tracked event, every exception surfaces as an Opportunity. We use the same fact model to track the build as we use to run client operations.
What the Tracking System Does

Pre-Deployment — Project Health Visibility

Tracks discovery completion, build guide progress, and Genie specification status across all clients and Mini-OS domains. Engineering velocity: which Genies are specced, built, tested, and handed to Codex.

  • Build guide completion % per client
  • Discovery status per Mini-OS
  • Genie pipeline: Specced ? Built ? Tested ? Deployed
  • Handoff readiness gates

At Handoff — Implementation Facts

When Codex and Azure AI execute the click-by-click build guides, every step completion fires a fact. Pass/fail gates. Real-time visibility on whether implementation is proceeding or stalled. Automated where pipeline hooks exist, manual fallback where not yet wired.

  • Step completion events from Codex pipeline
  • Azure AI Foundry deployment confirmations
  • Service Bus, Key Vault, Fabric provisioning events
  • UAT pass/fail per agent and stream

Post-Deployment — Live Operations Maturity

Once agents are live, the tracker shifts to maturity measurement. Digital Worker % creeping up over time. COGS reduction accumulating. Genie uptime, fact throughput, exception rates. The compounding ROI story — visualised per client, per Mini-OS, per Genie. This is the data that tells the fund story.

Primary operational store. Every implementation fact, maturity score, agent status, and system opportunity record lives here. Structured, relational, ISO 27001-compliant. Each client has their own Dataverse environment — data never commingles between clients.

EVENTS · 03

Automated trigger source
The automated fact source. When Codex executes a build step, when an Azure AI agent deploys, when a Power Automate flow activates — Service Bus topics carry those events to the fact writer. Decoupled, reliable, retry-enabled. Manual fallback writes direct to Dataverse.
AUTOMATE · 04
Power Automate
Orchestration · Notifications · Escalations
Orchestrates tracking workflows. Receives Service Bus events, writes facts to Dataverse, triggers Power BI refresh, fires escalation notifications when exceptions are raised, sends maturity milestone alerts. The connective tissue between event sources and the data store.
IMPLEMENT · 05
Codex + Azure AI Foundry
Implementation execution layer
Executes the click-by-click build guides. Each step maps to a Codex task and fires a Service Bus event on completion. When Azure AI Foundry deploys an agent, that deployment event is the fact. The implementation system and tracking system share the same data model.
MANUAL · 06

Fallback · Pre-automation phases
Before pipeline hooks are wired, engineers record facts manually — step completions, test results, go-live confirmations. The manual entry interface mirrors the automated fact schema exactly, so when automation takes over, the data model is already correct and history is complete.

Per-Client Isolation

Each Client Gets Their Own Instance

The tracking system is not a shared platform. Every client has a dedicated Dataverse environment, dedicated Power BI workspace, and dedicated Service Bus namespace. Their implementation facts, maturity scores, and operational data are isolated by design — ISO 27001 data classification and access control requirements demand it.

The NEIT MD overview dashboard aggregates across all client instances pulling only summary metrics (maturity %, Genie count, last fact timestamp) — never raw client data — into a portfolio view. Investors see an aggregated narrative that cannot be traced back to individual client operations.

Data Model · Implementation Events

Fact Stream Design

Every implementation step, agent deployment, test result, and go-live event is captured as a structured, immutable, timestamped Fact. This tab defines the fact types, payload schemas, trigger sources, and retention rules that govern the tracking system's data model.

?
The same model, used twice. RTBoss operational facts are business events — ORDER_RECEIVED, PARCEL_DISPATCHED. RTBoss tracking facts are implementation events — GENIE_DEPLOYED, TEST_PASSED, GO_LIVE_CONFIRMED. The schema is identical. The context differs. The discipline is the same.

Three-Lens View · Digital Worker Trajectory

Workforce Maturity

Maturity is the percentage of a client's operational workload handled by Digital Workers — agents — versus Hybrid (human + agent) or Human-only. The model tracks this across three lenses: per client, per Mini-OS domain, and per Genie. The compounding effect over time is the RTBoss commercial story.

?
The workforce maturity clock starts at MINI_OS_LIVE. From the moment a Mini-OS goes live, every maturity fact accumulates. The 30/90/180-day review cadence captures the progression. The target trajectory — 0% ? 18% ? 40% ? 80% Digital Worker over 24 months — is the measurable outcome RTBoss is sold on delivering.

How Workforce Maturity % Is Calculated

The Calculation Method

Digital Worker % = (Tasks completed autonomously by agents) ÷ (Total tasks that entered the system) × 100

A task that a Genie completes from trigger to output without human intervention counts as Digital. A task that requires a human decision or override at any point counts as Hybrid. A task that never touches an agent is Human.

The denominator is every task that entered the operational system in the measurement period. The numerator is every task resolved end-to-end by agents. Tasks that are escalated to humans and then returned to agents for completion count as Hybrid — even if agents did 90% of the work.

This definition is deliberately conservative. It means the maturity % only rises when agents genuinely own an outcome end-to-end. It makes the number credible and defensible — to clients, to the MD, and to investors.

Continuous Improvement · Pattern Detection

System Opportunities

As agents go live and facts accumulate, patterns emerge. Recurring exceptions, bottlenecks, underused Genies, missing fact types, handoff failures. The System Opportunities engine surfaces these signals as structured, actionable improvements — each with an owner, proposed fix, and link to the triggering facts.

?
The tracking system improves itself. Every GENIE_EXCEPTION fact, every BUILD_STEP_COMPLETED with status FAIL, every cluster of similar exceptions — these feed the System Opportunities engine. The system doesn't just record what happened. It tells you what to fix next.
The Five Opportunity Types
Exception Cluster
Recurring agent failure pattern
When the same GENIE_EXCEPTION fact type occurs more than 3 times in a 7-day window from the same Genie, an Exception Cluster opportunity is raised automatically. Payload includes the exception code, frequency, and linked fact IDs.
Auto-detectedHigh priority
Bottleneck
Step or Genie slowing throughput
When average processing time for a specific step or Genie exceeds 2× the baseline established in the first 30 days of operation. Indicates a prompt engineering issue, a tool latency problem, or an integration degradation.
Auto-detectedPerformance
Coverage Gap
Workflow reaching human when it shouldn't
When a human is overriding or handling a task type that appears in the same-pattern fact stream 10+ times per week — indicating a Genie should exist for this task but doesn't. Raised manually by engineers reviewing the hybrid task log.
Manual reviewMaturity lift
Missing Fact Type
Operational event not being captured
When engineers observe an operational event that has no corresponding fact type — meaning it's invisible to the tracking system and Digital Twin. Raised manually. Requires fact type definition, schema design, and trigger wiring before it can be automated.

Build process can be faster
When the time between GENIE_SPECCED and GENIE_DEPLOYED exceeds the rolling average by 50%+ — indicating a block in the build or implementation process. Also raised when a Genie type is built that matches an existing spec closely enough to reuse most of its components.

Exception Lifecycle

Every System Opportunity — whether auto-detected or manually raised — follows the same five-stage lifecycle. This ensures no opportunity is raised and forgotten. Each stage transition creates a new linked fact in the Fact Stream.

Opportunity Backlog Design

Where Opportunities Live

System opportunities are stored as records in Dataverse (neit_Opportunity table) and surfaced in a dedicated Power BI report page. Each system opportunity has: type, title, client, Mini-OS, severity, owner, lifecycle stage, linked fact IDs, and created/updated timestamps.

The backlog is reviewed at every 30/90/180-day workforce maturity review. High-severity system opportunities (Exception Cluster, Bottleneck) trigger immediate engineer review via Power Automate notification — they do not wait for the scheduled review cadence.

Closed system opportunities feed directly into the ISO 9001 Cl.10.3 Lesson Capture process in Section 09 · Audit Standards). Every improvement cycle makes the next deployment better.

Model Strategy Build vs Runtime Foundry Architecture

The Two-Layer Principle

RTBoss operates on a strict separation between construction time and runtime. These are not concurrent activities they are sequential, use different models, and serve completely different purposes. Conflating them is the most common architectural mistake in multi-agent systems.

Layer 01 Construction Time
GPT-5.3-Codex
Your engineer. Exits at deployment.

Codex 5.3 writes the system not a deployed agent within it. It generates Foundry Agent Service registration code for all 13 agents from the RTBoss spec, writes Azure Function and Logic App tool bindings, and produces the full test harness for Phase 10 validation scenarios. With agent-teams mode, multiple Codex sessions work in parallel across agents.

Codex 5.3 Responsibilities
-> Generate all 13 agent registration payloads
-> Write Cosmos DB + Fabric + Service Bus connectors
-> Scaffold orchestrator routing logic
-> Build Phase 10 automated test scenarios
-> Version, redeploy & extend on spec change
-> Collapses 11-phase build from days -> hours
Layer 02 Runtime Every Order Every Day
Opus 4.6 + Sonnet 4.5
Your workforce. Never exits.

Once deployed, Codex 5.3 steps out entirely. The production loop is owned by two Claude models with a deliberate cost/capability split Opus 4.6 at the orchestration layer where deep planning earns its premium, Sonnet 4.5 across all 13 specialists where bounded, structured work runs reliably at a fraction of the cost.

Runtime Model Assignment
Opus 4.6 -> RTBossOrchestratorAgent only
Sonnet 4.5 -> All 13 specialist agents
Sonnet 4.5 -> CutoffOrchestratorAgent
Haiku 4.5 -> High-volume lookup & validation
-> ~1 Opus call per order loop
-> ~12 cost reduction vs all-Opus approach
Strategic & Philosophical Differences Opus 4.6 vs Codex 5.3
Anthropic Opus 4.6
"Deep Thinker First"

Opus 4.6 is built on a reliability-before-speed doctrine. Its core architectural bet adaptive thinking means the model decides when to reason deeply and when not to. This is Constitutional AI heritage made practical: safety and trustworthiness are baked into the model, not bolted on. It is optimized less for one-shot cleverness and more for robust, multi-turn, tool-using workflows.

Why this matters for RTBoss
A pharmacy EMS making real dispatch decisions requires a model that checks its work, honours context across a long loop, and fails safely. Opus 4.6's caution is a feature, not a limitation. Its 1M token context holds an entire day's order history without degradation.
OpenAI Codex 5.3
"General-Purpose Builder"

Codex 5.3 merges the previous GPT-5.2 reasoning model and the GPT-5.2-Codex coding agent into a single system. Its philosophy is end-to-end execution not just writing functions, but understanding all the work around the code: updating Jira, writing documentation, managing deployment pipelines. How Foundry Wires This Together Three Primitives
01 Connected Agents
Primary -> Sub-agent Delegation

The Opus 4.6 Orchestrator is the primary agent. The 13 Sonnet 4.5 specialists are connected sub-agents. Foundry manages context handoff, retries, and routing no hand-coded routing logic required. This is the DOL pattern implemented natively in the platform.

02 Workflows
Stateful Long-Running Loops

The context envelope the Digital Twin state lives in the Workflow layer between agent calls, not only in Cosmos DB. Built-in error handling, retries, and recovery replace most of the Service Bus dependency for inter-agent handoff. The full order loop is a single stateful Workflow execution.

03 Foundry IQ
Unified Grounding API

DrugLookupAgent currently hits a Fabric Delta table directly. Foundry IQ becomes the abstraction layer querying Fabric, OneLake, and drug reference data through one governed, permission-aware API. Reduces hardcoded connectors and respects data classification boundaries automatically.
