Execution Modes - Who Executes Work

Work is classified by who or what executes it. Classification determines routing, automation potential, and SLA. It is not static - work progresses from Human -> Hybrid -> Digital as the Agentic Factory is built out.

Digital Worker

Fully autonomous. Agent handles end-to-end with no human in the loop. Deterministic, high-confidence, in-policy.

Execution Pattern

Fact arrives -> Genie executes OR Stream runs -> Outcome logged to Digital Twin -> No human involved

Examples
  • Password reset (Genie auto-executes)
  • Patch deployment (Stream orchestrates)
  • Invoice under auto-approval threshold
  • Scheduled report generation
AutonomousNo human loopHigh volume

Hybrid Worker

Agent does the heavy lifting, human provides judgment or approval at the gate. True collaboration between digital and human.

Execution Pattern

Fact arrives -> Genie/Stream prepares work -> Surfaces decision to human -> Human decides -> Agent completes -> Outcome logged

Examples
  • Expense over threshold (agent validates, human approves)
  • Complex troubleshooting (agent diagnoses, human decides)
  • Contract review (agent flags risks, human confirms)
  • Ambiguous classification (agent scores, human chooses)
Agent + HumanDecision gateQuality control

Human Worker

Human drives execution end-to-end. Requires expertise, creativity, relationships, or strategic judgment that cannot be systematised.

Execution Pattern

Fact arrives -> Assigned to human -> Human executes (may invoke Genies for research or drafting support) -> Outcome logged

Examples
  • Client escalation requiring empathy
  • Architectural design decisions
  • Sales negotiation
  • Strategic planning
  • Novel problem-solving
Human-ledAgent supportHigh judgment

Key principle: Classification determines execution path, not arrival pattern. An urgent email might be Digital Worker (password reset) or Human Worker (CEO escalation). Work type is about who executes, not when it arrived or how urgent it is.

Section 02 - OS Framework - RTBI
Section 02 Flow

RTBI - Real-Time Business Intelligence

RTBI is not a reporting tool bolted onto operations - it is what operations look like when they run correctly on RTBoss. Two foundational components (Unified Data and the Digital Twin) make three intelligence components possible (Planning, Analytics, Decisions). You cannot have the top without the bottom.

Foundation: Unified Data - Digital Twin -> these make possible -> Intelligence: Planning - Analytics - Decisions Every metric, forecast, and recommendation is a direct expression of live Fact data - not a report compiled after the fact.
Foundation Layer - The Infrastructure Intelligence Runs On

These two components are not intelligence outputs - they are the substrate that makes intelligence possible. Unified Data ensures every component works from the same live numbers. The Digital Twin ensures every decision has a complete, replayable history behind it. Both must exist before Planning, Analytics, or Decisions mean anything.

?
Unified Data
Single source of truth - No ETL - Live

One data layer across all Mini-OS instances. Every Fact from every domain - finance, operations, logistics, HR - flows into a single governed, real-time fabric. There is no reconciliation process because there is no second copy. One number, updated the moment the Fact occurred.

Without this, every downstream intelligence component is working from different versions of reality. Unified Data is the reason RTBI can be trusted.

PlatformMicrosoft Fabric + Dataverse
ETL requiredNone

Single truthNo silosReal-time fabric
Digital Twin
Complete history - Replay - Safe testing

The complete, append-only record of everything that has happened in the business - every Fact, every agent decision, every asset mutation, every dimension change - in sequence, with timestamps, forever. Not a snapshot. Not a summary. The full story.

This means two things that traditional systems cannot do: replay any moment in business history to understand exactly why an outcome occurred, and test any proposed change against real historical data before it touches production. Zero production risk on every process change.

Intelligence Layer - What the Foundation Makes Possible

These three components are the intelligence outputs - the reason you build the foundation. They move in sequence: Analytics describes what is happening now, Planning uses that to project forward, and Decisions acts on both. Each is only as good as the data and history underneath it.

Real-time dashboards and KPI tracking running directly on the Unified Data layer. Every metric updates the moment a Fact occurs - no refresh cycle, no manual pull. The end of self-reporting. No one emails a status update because the status is live, visible, and accurate at all times.

Event Stream vs Fact Stream - Critical Distinction

Not Every Event Is a Fact

Just because something happened doesn't mean it needs to become a Fact in the RTBoss loop. Without this filter, the Fact stream would be overwhelmed with noise - millions of routine events that require no action. We use a two-tier architecture to separate signal from noise.

Event Stream (Tier 1)

Everything that happens. High volume, mostly routine. Stays in source systems (Zero Touch AI, UniFi, Azure Monitor, access control databases).

Examples (NOT Facts)
  • Server receives 10,000 log entries/minute
  • Access control logs 500 badge swipes/day
  • Network heartbeat every 30 seconds
  • Temperature sensor reading every 5 minutes (within range)
  • Successful backup completion (expected)
High volume Routine No action needed

? Fact Stream (Tier 2)

Events that require attention. Low volume, high signal. Enters RTBoss loop, gets Dimensions, routed to Workers.

Examples (ARE Facts)
  • Server CPU >90% for 5 minutes -> resource_threshold_exceeded
  • Badge denied 3 times -> access_denied_repeated
  • Device heartbeat missed 10min -> device_offline
  • Temp outside range -> environmental_alert
  • Invoice received -> invoice_received
Low volume Actionable Requires decision

The Promotion Rule: Event -> Fact

An Event becomes a Fact when it meets ANY of these criteria:

  • Threshold crossed - metric outside acceptable range (CPU >90%, disk <10%, response time >2s)
  • Pattern detected - repeated failures, anomaly, unusual sequence (3 failed logins, traffic spike)
  • State change - transition from one state to another (online -> offline, pending -> approved)
  • Business event - transactional event requiring processing (invoice received, order placed, payment due)
  • Human input - any human-generated communication or request (email, call, form, chat, meeting action)
?
Anti-pattern: Do NOT promote routine operational events to Facts. A successful login is not a Fact. A normal temperature reading is not a Fact. A heartbeat ping is not a Fact. These stay in their source systems for compliance, forensics, and historical analysis - but they don't enter the RTBoss loop.

Dimension Enrichment - Adding Context to Facts

Traffic Light System - Simple, Objective Classification

Every Fact gets a Traffic Light based on one simple question: "Is someone blocked from doing their job?" No judgment calls. No debating P1 vs P2. Just: blocked, workaround, or routine.

RED - BLOCKED

Someone is completely unable to work RIGHT NOW. No workaround exists.

Examples
  • CEO completely locked out (all devices)
  • Prod database offline
  • Payment gateway down (only payment method)
  • Client's only contact number offline
  • Internet connection down (no backup)
Action

Emergency - interrupt work, page on-call if needed, escalate immediately

YELLOW - WORKAROUND

Degraded or broken, but temporary workaround exists. Business continues inefficiently.

Examples
  • Boardroom AV broken (use laptop screen share)
  • Desktop Outlook broken (Outlook Web works)
  • One printer offline (others available)
  • Network degraded to 50% speed
  • Mobile app broken (desktop works)
Action

Priority - fix today, top of queue, escalates to RED if workaround fails

GREEN - ROUTINE

Normal operations, no one impacted. Routine work, scheduled tasks, requests.

Examples
  • Client inquiry email
  • Feature request
  • Scheduled migration (no users yet)
  • Password reset (user on vacation)
  • Documentation update
Action

Routine - process in queue order, auto-escalates based on timer

Decision Tree - Simple Classification

Is anyone unable to do their core job function?
+- YES -> Can they use a workaround?
+- YES -> YELLOW (degraded but functioning)
+- NO -> RED (completely blocked)
+- NO -> GREEN (routine work)

Escalation Flow - Facts Change Color

Facts can change Traffic Light as situations evolve or time passes:

GREEN -> YELLOW
  • System starts degrading
  • Workaround becomes available
  • Example: Printer fails -> tech enables backup printer nearby
YELLOW -> RED
  • Workaround stops working
  • Too many people affected (workaround doesn't scale)
  • Example: Backup printer also fails -> now RED
GREEN -> RED (skip YELLOW)
  • Escalation timer expires
  • Scheduled deadline hits
  • Example: Boardroom AV broken (GREEN) -> meeting starts in 5min -> RED
RED -> YELLOW
  • Workaround implemented after initial failure
  • Partial restoration of service
  • Example: Primary internet down (RED) -> mobile hotspot working (YELLOW)

Company Cadence - Agentic Operating Rhythm

The regular rhythm that keeps New Era IT synchronized - but reimagined for an agentic world. Automated summaries replace status meetings. Humans focus on decisions, learning, and strategy. The system tells us what happened; we decide what to do about it.

Company Cadence := the fixed schedule of automated summaries and decision meetings. Daily digests. Weekly retrospectives. Monthly business reviews. Quarterly planning. Annual strategy. No manual status reporting - the system knows what's happening in real-time.
?
Philosophy shift: Traditional standups exist because humans need to manually synchronize status. But in RTBoss, the system already knows what everyone is working on (Facts assigned), what's been completed (Facts closed), and what's blocked (Blocker dimension). So we don't have status meetings - we have automated summaries + human decision meetings.
Daily Cadence - Automated Digest + Exception Sync

Morning: Automated Daily Digest (No Meeting)

What

Automated summary of current system state delivered to Teams channel + email. Generated by Daily Summary Genie (Hybrid Worker).

When & Where

Time: 8:30 AM AEST (before work starts)
Delivered to: Teams "Daily Digest" channel + email

Digest Contents

  • RED Facts - All currently blocked facts (count, assigned to, age)
  • YELLOW Facts escalating soon - Facts that will become RED today if not addressed (with countdown timers)
  • ? Closed yesterday - Count by Worker Type (Digital/Hybrid/Human)
  • Blockers requiring coordination - Facts tagged with blockers that need team help (not just "waiting on external")
  • Trends - Pattern detection (e.g., "password resets up 40% this week", "3 network slow reports from same location")

?
No meeting required. Team reads digest async. If you see something urgent, ping in Teams or grab the fact. Otherwise, just awareness of current state.

Trigger

Format

Duration: 10 minutes max
Location: Teams call

Agenda

  2. Assign coordination actions (who unblocks whom)
  3. Done

Rule: If no coordination blockers in the digest -> no meeting that day. Team just reads digest and gets to work.

What Gets Automated vs What Stays Human

? AUTOMATED (Genies Do This)

  • Status compilation - what's open, closed, blocked
  • Metrics calculation - resolution times, volumes, distributions
  • Pattern detection - repeat issues, trends, anomalies
  • Opportunity identification - client requests, automation candidates
  • Report generation - daily digest, weekly summary, monthly review
  • Client health scoring - RED/YELLOW/GREEN based on fact patterns

HUMAN (We Decide This)

  • Lessons learned interpretation - what do patterns mean?
  • Automation roadmap prioritization - which Genie to build next?
  • Strategic client decisions - proactive outreach, escalation handling
  • Resource allocation - who works on what, hiring decisions
  • Direction setting - pivots, new bets, strategic focus
  • OKR definition - what are we committing to achieve?
?
The agentic cadence principle: Humans don't compile information - agents do. Humans don't report status - the system does. Humans make decisions, learn from patterns, and set direction. The cadence exists to create regular decision points, not status updates. If a meeting is just reporting what everyone already knows, automate it.
